ScholarAIO Demo Plus

这是升级版的系统演示图。相比 demo.html，这个版本把内容拆成三层：先看全局泳道图，理解“谁发起、谁编排、谁执行、谁落盘”；再看四条高频任务时序，理解不同任务是怎么穿过 skill / CLI / MCP / Python 核心模块；最后看入口对照表，把 Claude Code、CLI、MCP 和插件模式的职责边界一次讲清楚。

总览：泳道图 细节：任务时序 对照：入口速查表 重点：skill / MCP / LLM / embedding 分工

一、全局泳道图

横向看职责，纵向看流向。你可以把它理解成 ScholarAIO 的“系统总线图”。

用户与客户端 编排层：skill / CLI / MCP Python 核心能力层 数据与索引 外部能力与服务

二、四条高频任务时序

这部分专门回答“一个具体任务发起后，系统内部到底发生了什么”。

场景 A：在 Claude Code 里说“帮我找 2020 年后的拖曳减阻论文”

要点：Claude Code 场景先过 skill，再落到 search / vectors

场景 B：CLI 执行 scholaraio pipeline ingest

要点：CLI 最适合排障，因为链路最直

场景 C：Claude Code 里说“分析主题分布并写综述”

要点：写作 skill 通常建立在 search / show / workspace / topics 之上

场景 D：外部 MCP 客户端调用 show_paper

要点：MCP 是暴露层，不是另一套业务实现

三、入口与场景速查表

这张表适合你实际使用时快速判断：我现在应该走哪种入口、底层会怎么流、我该期待什么结果。

入口 适合的用户动作 中间机制 最终调用 最适合的场景
Claude Code 自然语言说“帮我找论文”“帮我总结”“帮我写综述” 先匹配 skill，再由 agent 组织 CLI / 文件 / 推理步骤 search / show / pipeline / workspace / topics 等核心模块 研究协作、探索、写作、综合任务
CLI 直接执行 scholaraio 子命令 cli.py 解析参数并分发 直接命中 index.py、loader.py、pipeline.py、vectors.py 调试、批处理、环境排障、最小闭环验证
MCP 客户端 在外部客户端里调用 search / show_paper / build_index 等 tool mcp_server.py 封装为标准 tool 接口 仍然走同一套 Python 模块 IDE 集成、跨客户端调用、图形界面接入
插件模式 在任意项目里安装 ScholarAIO plugin SessionStart hook -> check-deps.sh -> 全局配置初始化 最终仍落到 scholaraio CLI / package / MCP 跨项目长期使用、快速接入技能市场

四、最关键的理解

Skill 是编排层

它决定“什么时候该做什么”，但不是真正执行 search / ingest / topics 的地方。

CLI 是最直的入口

如果你怀疑系统有问题，优先先跑 CLI，因为最容易定位到底卡在哪一层。

MCP 是接入层

它把已有能力封装给外部客户端，不应被理解成另一套并行系统。

外部能力分工不同

LLM 负责提取/判断，Embedding 负责语义向量，MinerU 负责 PDF 转 Markdown，三者职责完全不同。

这个版本保留了 demo.html 的全局认知优势，但把“系统图”和“任务时序”拆开了，更适合你反复对照使用。建议的阅读顺序：先看全局泳道图，再看四条时序，最后看入口速查表。
