ScholarAIO 使用流 Demo

这张图把 ScholarAIO 当成一个完整系统来看：用户可以从 Claude Code、CLI、MCP 客户端等不同入口进入，发起搜索、读论文、入库、写作、主题分析等任务；任务会经过 skill / 命令 / MCP tool 这一层，再流入 Python 核心模块，最终落到知识库、索引和工作区数据上。

入口层：Claude Code / CLI / MCP 中间层：Skill / CLI 子命令 / MCP Tools 能力层：pipeline / loader / index / vectors / topics 数据层：papers / index.db / topic_model / workspace

一、全局流动图

用户入口 编排层 Python 核心模块 数据/索引落点 LLM / Embedding / MinerU 外部能力

二、按场景看的 6 条 Demo 流

场景 1：Claude Code 里“帮我找相关论文”

  1. 用户在 Claude Code 输入自然语言请求。
  2. Agent 根据描述匹配 search skill。
  3. Skill 引导调用 search / vsearch / usearch。
  4. 底层流向 index.py 或 vectors.py。
  5. 返回论文列表，再由 agent 组织成对话回复。

场景 2：CLI 里直接检索

  1. 用户执行 scholaraio search 或 vsearch。
  2. cli.py 解析参数。
  3. search 走 SQLite FTS5；vsearch 走 embedding + FAISS。
  4. 命中结果后直接打印到终端。

场景 3：把 PDF 入库

  1. 用户把 PDF 放入 data/inbox。
  2. 执行 scholaraio pipeline ingest 或通过 ingest skill 触发。
  3. pipeline 调 MinerU，把 PDF 转成 Markdown。
  4. extractor / metadata 用 regex + LLM + API 富化元数据。
  5. dedup 后写入 data/papers，再更新 index / embed。

场景 4：读论文某一层内容

  1. 用户说“给我看这篇论文摘要/结论/全文”。
  2. show skill 或 CLI show 命令命中 loader.py。
  3. loader 根据 layer 读取 meta.json 或 paper.md。
  4. L1-L4 渐进返回，避免一开始灌全文。

场景 5：做主题分析或研究综述

  1. 用户要求“分析主题分布”或“帮我写综述”。
  2. topics / literature-review / workspace skill 参与编排。
  3. topics.py 复用已有 embeddings 做 BERTopic。
  4. workspace 缩小论文集合，再交给写作 skill 组织输出。

场景 6：MCP 客户端调用

  1. 外部客户端通过 MCP 请求 tool。
  2. mcp_server.py 中对应工具接收参数。
  3. 工具内部调用同一套 Python 核心模块。
  4. 返回结构化 JSON 给客户端，再由客户端展示。

三、最容易混淆的三个点

Skill 不是能力本体

Skill 更像工作流说明书。它告诉 agent 什么时候该调用 search、show、pipeline、workspace，但真正的实现仍然在 Python 模块和 CLI / MCP 层。

MCP 不是另一套业务逻辑

MCP 只是把已有能力打包成标准 tools 给外部客户端用。CLI 跑得通，MCP 才容易稳定；CLI 没跑通，MCP 只会换个入口报错。

LLM / Embedding / MinerU 分工不同

LLM 负责提取、判断、富化；Embedding 负责语义检索和主题建模；MinerU 负责 PDF 转 Markdown。三者都是外部能力，但职责完全不同。

这份 demo 更适合先建立全局认知，再回去对照 README、CLAUDE.md、explain.html 和 explain.md。若你需要，我下一步可以继续补一版“时序图风格”的 demo，把一个具体请求从用户输入到 index.db / paper.md 落盘的每一步画得更细。
